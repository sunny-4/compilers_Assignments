Part 6: Additional Information
Project: CS327 LALR(1) Parser Assignment

1) Implementation Highlights
----------------------------
- Two parser tracks are maintained for assignment workflow:
  - `lalr_assignment3.y` (intentionally unresolved grammar for conflict demonstration)
  - `lalr_assignment3_resolved.y` (conflict-resolved grammar for final parser behavior)
- Conflict-resolved grammar maintained in: `lalr_assignment3_resolved.y`
- Matching lexer maintained in: `lalr_assignment3_resolved.l`
- Parser features integrated beyond basic parsing:
  1. LALR(1) parsing table matrix output (ACTION/GOTO style)
  2. Reduction logging during parse
  3. Reverse derivation display after successful parse
  4. Improved diagnostics with source context and fix hints

2) Quantitative Parser Analysis
-------------------------------
- Original grammar (`lalr_assignment3.output`):
  - 363 shift/reduce conflicts
  - 0 reduce/reduce conflicts
  - representative conflict families:
    - expression/operator ambiguity (states 139–157 each with many SR conflicts)
    - dangling-else ambiguity (state 184)
- Resolved grammar (`lalr_assignment3_resolved.output`):
  - no `conflicts:` entry (i.e., no reported SR/RR conflicts)
- Number of states in resolved automaton:
  - 224 states (`grep -c "^State [0-9]" lalr_assignment3_resolved.output`)

3) Error-Diagnostic Strategy
----------------------------
Diagnostics are designed to be practical for debugging source programs:
- precise error location: line + column
- unexpected lexeme reporting
- source-line print with caret marker (^)
- grammar-aware hints (missing `;`, missing `:`, unmatched `else`, early EOF, etc.)

This improves usability over plain "syntax error" output.

4) Testing and Validation Notes
-------------------------------
- A dedicated test suite exists under `tests/`:
  - `tests/valid/` for accepted programs
  - `tests/invalid/` for rejection and diagnostic checks
  - `tests/conflict_examples/` for dangling-else and related behavior
- Validation approach used:
  - compile unresolved and resolved parser/lexer flows separately
  - run representative valid and invalid files
  - verify ACCEPTED/REJECTED status and report artifacts
  - compare conflict summaries from `lalr_assignment3.output` vs `lalr_assignment3_resolved.output`

5) Design Choices and Rationale
--------------------------------
- Conflict-resolution stage uses precedence (`%precedence IFX`, `%precedence ELSE`) rather than major grammar refactor.
  - Benefit: small localized change, preserves grammar readability.
- Conflict-demonstration stage intentionally keeps an ambiguous expression grammar in
  `lalr_assignment3.y` so Part 1 can show real SR conflicts before resolution.
  - Benefit: clear before/after evidence for assignment requirements.
- Parsing table is generated by parsing the Bison `.output` automaton text.
  - Benefit: table reflects actual generated parser states/actions.
- Reverse derivation is built from reduction sequence storage.
  - Benefit: directly aligned with bottom-up parse process.

6) Current Limitations
----------------------
- Diagnostics use heuristic hints; they are helpful but not full semantic recovery.
- Parsing table output is wide due to many symbols/states.
- Grammar-level acceptance does not perform semantic checks (types, declarations, etc.).

7) Suggested Future Extensions
------------------------------
- Add panic-mode recovery rules for multiple-error reporting in one run.
- Split parsing-table print into ACTION and GOTO sections for compactness.
- Add semantic analysis layer (symbol table, type checks, undeclared identifier checks).
- Add a small script to run all tests and produce pass/fail summary automatically.

8) Reproducibility Notes
------------------------
Typical workflow:
1. Generate parser (`bison`) and scanner (`flex`)
2. Build executable (`gcc ... -lfl`)
3. Run executable with input path and optional report path
